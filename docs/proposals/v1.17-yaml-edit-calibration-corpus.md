# v1.17 yaml_edit signals — calibration edge case corpus

> Pre-Monte Carlo design memo. Lists edge cases the eventual MC scenario
> generator MUST cover before tuning the 8 yaml_edit weights against
> ground truth. Compiled from the kashealth E2E pilot run + general
> reasoning about SAFe LPM lifecycle states.

The 8 yaml_edit weights ship at conservative round-number defaults
(see `cw_heuristics.yaml.tmpl § yaml_edit_weights`). Calibration is
deferred to the first kashealth PI close, where:

1. PM provides per-person hour ground truth (gut estimate)
2. Engine output is compared via MAD vs ground truth
3. `/edpa:calibrate` autocalib loop tunes weights to minimize MAD

**Before** tuning, the MC generator must produce scenarios that exercise
each edge case below — otherwise weights end up optimised for the
average case but break on the long tail.

## 1. Item lifecycle edge cases

### 1.1 Initiative full lifecycle (Funnel → Done across 5 PIs)
- Created in PI-N as Funnel
- Reviewed in PI-N+1 → Reviewing
- LBC authored in PI-N+1 (block_add: business_case, block_add: benefit_hypothesis)
- Risks added in PI-N+1.3 (list_grow × 3-5)
- Decomposed into 2-4 Epics in PI-N+1.4 (decomposition_sketch)
- LBC approved → Ready in PI-N+2
- Implementing across PI-N+2..PI-N+3
- Done in PI-N+3 (with 4-8 child Epics done)
- **Calibration target**: total credit per person across the lifecycle
  matches their actual share (PM/Arch typically own ~60-80% of LBC,
  Devs add elaboration in PI-N+1.4)

### 1.2 Epic compressed lifecycle (single PI)
- Created Funnel → Done in same PI (small Epic, e.g. enabler)
- All gates fire within 5 iterations
- **Edge**: Epic with js=3 but 8 status transitions ⇒ gate_events
  produce small per-event credit; yaml_edit signals dominate

### 1.3 Defect open + close in IP iter
- D-N created in PI-N.5 (yaml_edit:create)
- D-N status: Funnel → Implementing → Done (3 transitions)
- Bug A fix means Defect is now loaded as Done item (level=Defect)
- **Edge**: Defect both gets Story-style Done credit AND yaml_edit
  signals. Calibration must verify no double-counting.

### 1.4 Story rejected → reopened
- S-N status Backlog → Implementing → **rejected** (Validating → Backlog)
- Re-implemented PI-N+1
- **Edge**: revert signal might fire incorrectly; net diff over multi-PI
  span needs to handle.

### 1.5 Task vs Story routing
- Task = "technical work without user-facing value" per edpa.yaml
- Currently flowed through same Story credit path
- **Edge**: Task should match Story behavior (Done + yaml_edit signals)
  but probably warrants separate calibration weight if MAD shows
  systematic over-credit on tasks.

## 2. Authoring pattern edge cases

### 2.1 LBC author vs reviewer (single-author LBC commit)
- Urbánek writes full business_case + benefit_hypothesis + leading
  indicators in ONE commit (block_add ×3, scalar ×N, list ×3+ for
  indicators)
- Tůma later adds /contribute @sirTurbisCZ weight:2.0 in PR review
- **Calibration target**: Urbánek ~70%, Tůma ~30%

### 2.2 Sequential AC authoring (incremental)
- Day 1: AC-1 added by Turyna (1 list_grow signal, weight 1.0)
- Day 2: AC-2 added by Matoušek (1 list_grow signal, weight 1.0)
- Day 3: AC-3, AC-4 added by Turyna (2 list_grow signals)
- **Calibration target**: Turyna ~75%, Matoušek ~25% (relative weight share)

### 2.3 Pair-edit single commit
- Squash merge with `Co-authored-by: A` and `Co-authored-by: B`
- Primary author gets yaml_edit signal credit
- /contribute trailers in commit message redirect partial credit
- **Edge**: yaml_edit signal goes to commit author; /contribute
  signals go to listed names. Calibration verifies both reach the
  same final cw split as a non-squashed multi-commit PR.

### 2.4 Hackathon spike
- One person writes 8 hours of LBC + AC + NFRs across 12 commits in
  one day (e.g., Friday before PI close)
- Total yaml_edit weight: ~50-80 raw points
- **Edge**: no per-day cap (per design); engine normalizes via
  capacity bound (max 40h/iter). Calibration verifies hackathon
  doesn't distort other team members' credit shares.

### 2.5 Bot-assisted commits
- Operator runs `python3 scripts/decompose_epic.py E-5`
- Tool generates F-7, F-8, F-9 stubs and commits as the operator
- These commits look identical to manual creation
- **Edge**: cannot reliably distinguish from human work; calibration
  must accept some bot-assisted credit as legitimate (the operator
  ran the tool, that IS work).

## 3. File-system edge cases

### 3.1 Item rename / move
- Story originally under F-1 → moved to F-7 (parent: change)
- Single commit with `git mv` + content edit
- **Detection**: yaml_edit_signals.py skips `rename` lines in the diff
  header; only counts content delta. Verify no double-credit.

### 3.2 Item deletion + re-creation (id reuse)
- S-1 deleted in PI-N.3 (after triage rejection)
- New S-1 created in PI-N+1.1 with different content
- **Edge**: yaml_edit:revert fires for deletion; yaml_edit:create fires
  for re-creation. Both credited correctly (no merge).

### 3.3 YAML format-only churn
- `yaml.safe_dump` reorders keys differently between operator runs
- Diff shows entire file as -content +content with same semantic data
- **Detection**: structural signals don't fire because no NEW blocks/
  list items detected (set diff is empty). Verify in tests.

### 3.4 Initial bulk seed (`backlog.py add` × 30)
- Operator seeds 30 stories in one sitting (single committer)
- Each commit creates one new YAML
- 30 × yaml_edit:create = 150 raw weight to one person
- **Edge**: legitimate seed work, but compresses ~3 weeks of effort
  into one day. Calibration: per-iteration normalization handles this
  (max hours = capacity).

## 4. Multi-person edge cases

### 4.1 Strategic-role under-attribution
- Urbánek (Arch) does PI Planning prep — high-volume YAML editing
  but few code commits
- Turyna (Dev) writes most code but minimal backlog edits
- **Calibration target**: yaml_edit signals correctly attribute
  strategic work; Urbánek's iteration credit reflects ~40h of LBC +
  decomposition work, not 0h as in pre-v1.17 engine.

### 4.2 Pair-programming (two people, one keyboard)
- Pair-prog session produces 5 commits authored alternately by A and B
- Each commit's yaml_edit credit goes to its author only
- No /contribute directive; pair credit is implicit in commit alternation
- **Calibration target**: net credit ~50/50 between A and B over the
  session.

### 4.3 Mob-programming (3+ people, one keyboard)
- Same as 4.2 but with 3+ rotating drivers
- /contribute directives in commit messages credit all participants
- yaml_edit signal goes to commit author; /contribute additive

### 4.4 Onboarding ramp
- New team member added mid-PI
- First few commits are exploratory (e.g., S-N status reverts during
  hand-holding)
- yaml_edit:revert fires; net credit may be near-zero or negative
- **Calibration**: verify revert weight isn't so harsh that newbies
  get penalized for legitimate learning churn.

## 5. Time-window edge cases

### 5.1 Backdated commits
- Operator uses `git rebase -i` to clean up history before PR merge
- Original `GIT_AUTHOR_DATE` preserved; new commit timestamps differ
- **Detection**: yaml_edit_signals.py uses author date for window check
  (correct). Verify across rebase scenarios.

### 5.2 Cherry-pick across PIs
- Bug fix committed in PI-N.4
- Cherry-picked to release branch in PI-N+1.2
- New commit has new author date in PI-N+1
- **Edge**: original author gets credit in PI-N (original commit) AND
  cherry-picker gets credit in PI-N+1 (new commit). This is correct —
  both did real work (fix + port).

### 5.3 Commits at iter boundary
- Commit at 2026-05-17T23:45:00 (last 15 min of PI-N.1)
- Commit at 2026-05-18T00:30:00 (first 30 min of PI-N.2)
- **Detection**: window check is inclusive of start-date 00:00 and
  end-date 23:59:59. Verify: no commit can be credited to two iters.

### 5.4 Long-running PR spans 2 iterations
- PR opened 2026-05-15, merged 2026-05-22 (spans PI-N.1 → PI-N.2)
- Commits across both iterations
- **Detection**: each commit credited to its own iter via author date.
  PR-API signals (pr_reviewer, /contribute) credited at merge iter.

## 6. Calibration measurement edge cases

### 6.1 PM gut estimate vs reality
- PM estimates Urbánek 35h, Tůma 40h, Turyna 38h, Matoušek 32h
- Engine output: Urbánek 40h, Tůma 38h, Turyna 40h, Matoušek 35h
- MAD = mean(|diff|) = mean(5+2+2+3) = 3h ⇒ 7.5% of capacity
- **Acceptance**: MAD ≤ 15% of mean capacity (per kashealth pilot SC #3)

### 6.2 Sparse iteration (low activity)
- Holiday week, only 5 commits across team
- yaml_edit signals + gate_events both small
- Engine output: most people 0-10h, one person 30h
- **Edge**: capacity_override.py needs to be applied per absent person;
  calibration verifies leftover capacity isn't redistributed silently.

### 6.3 Iteration with no Done items
- All work in-progress, nothing reaches Done
- yaml_edit signals fire for elaboration work
- gate_events fire for transitions through non-Done states
- **Edge**: engine still produces hours from yaml_edit + gate_events,
  no reliance on Done-only path. Pre-v1.17 this iter would show 0h.

## 7. Calibration corpus structure

The MC generator should produce ~1000 synthetic PI-scale scenarios.
Distribution across edge case categories:

| Category | Scenarios | Why |
|---|---|---|
| Normal (1.1, 4.1) | 400 | Most common; tunes core weights |
| Defect lifecycle (1.3) | 100 | Validates Bug A fix doesn't over-credit |
| Hackathon / spikes (2.4) | 100 | Tunes lines_volume cap |
| Bulk migration (under section 1) | 50 | Validates discount factor |
| Pair / mob (4.2, 4.3) | 100 | Validates multi-author splits |
| Time-window edges (5.x) | 50 | Validates window filter |
| Sparse / no-Done (6.2, 6.3) | 100 | Validates non-zero output without Done items |
| Onboarding / reverts (4.4, 1.4) | 100 | Tunes revert weight |
| Item moves (3.1, 3.2) | 50 | Validates rename detection |
| Strategic-role (4.1) | 50 | Validates Urbánek-style high-elaboration credit |

Each scenario has known per-person ground truth (we generated it),
so MAD is computable. Calibration optimises 8 yaml_edit weights via
coordinate descent or random-sample search.

## 8. Acceptance criteria for v1.17 calibration

- MAD ≤ 0.10 across all 1000 MC scenarios
- MAD ≤ 0.15 on real kashealth pilot PI-1 (PM ground truth)
- No scenario where one person's credit exceeds capacity (engine
  invariant: derived ≤ capacity)
- No scenario where team total exceeds 100% of capacity
- yaml_edit signals contribute meaningfully (not dominated by
  pr_author/commit_author signals to point of irrelevance)
- Calibrated weights stay within ±50% of conservative defaults
  (large drift signals over-fitting)

## 9. Calibration artifacts

After calibration loop completes, ship:

- `tests/test_yaml_edit_signals.py` regression tests for all edge cases
