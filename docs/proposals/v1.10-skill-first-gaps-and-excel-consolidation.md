# RFC — v1.10.0 skill-first gaps + Excel workbook consolidation

- **Status:** **proposed** (not implemented)
- **Author:** Jaroslav Urbánek (scope decision 2026-05-07, post-pilot-prep review)
- **Target release:** v1.10.0
- **Required for:** kashealth pilot retro (post-PI-2026-1 close); not blocking pilot kickoff
- **Effort estimate:** ~250 lines code (3 skills + 1 engine refactor) + ~120 lines tests, ~2 dev-days

## 1. Why this RFC exists

Pilot kickoff prep (`docs/kashealth-pilot/KASHEALTH-PILOT.md`) revealed
two structural debts that didn't block 1.9.0 release but contradict
the **skill-first validation mandate** (memory:
`feedback_skill_first_validation.md` — *"skill flow je canonical
customer UX, ne script flow"*):

1. The kickoff runbook has 14 sections, of which 9 still call
   `python3 .claude/edpa/scripts/...` directly. That's not a
   documentation problem — it's a missing-skill problem. Three
   real gaps where current skills fail customer UX.
2. `summary.xlsx` and `item-costs.xlsx` are split across two
   `Workbook()` instances in `engine.py` for no domain reason —
   it's a code-shape artifact, not a design decision. One file
   with two tabs is the standard audit-deliverable pattern.

Neither item is urgent. Both belong in v1.10.0 as paying down
debt before scaling the pilot to PI-2026-2 / further customers.

## 2. Scope

### 2.1 Excel workbook consolidation

**Today:** `engine.py:1126` and `engine.py:1185` build two separate
`Workbook()` objects, written to `summary.xlsx` and
`item-costs.xlsx`.

**v1.10:** one `edpa-results.xlsx` with two tabs:

| Tab | Source | Content |
|-----|--------|---------|
| **Team Summary** | today's `summary.xlsx` | Person · Role · FTE · Capacity · Derived · Items · OK |
| **Item Costs** | today's `item-costs.xlsx` | Item · Level · JS · Person · CW · Score · Ratio · Hours |

**Sketch:**

```python
wb = Workbook()
ws_summary = wb.active
ws_summary.title = "Team Summary"
# … existing summary rows …
ws_items = wb.create_sheet("Item Costs")
# … existing item-cost rows …
wb.save(report_dir / "edpa-results.xlsx")
```

**Migration:**

- Drop both `summary.xlsx` and `item-costs.xlsx` from
  `reports.py` outputs and `.edpa/reports/iteration-*/` layout.
- Replace with single `edpa-results.xlsx`.
- Update fixtures in tests that count xlsx file presence (grep
  `summary.xlsx` and `item-costs.xlsx` in `tests/`).
- Update kickoff runbook § 0 ("Quick orientation") and § A
  ("Pass kritéria po setupu") to reference the new filename.
- Update `plugin/skills/edpa-reports/SKILL.md` output description.

**Backwards compat:** none. v1.10 is a minor bump, customers
re-run engine for new format. Pilot is on v1.9.0 at consolidation
time, so existing kashealth artifacts stay readable as historic
snapshots; only post-v1.10 iterations switch to consolidated file.

### 2.2 Skill-first gap closure

Three gaps where today's skills fail and the runbook had to
crutch them:

#### 2.2.1 New skill: `/edpa:preflight`

**Purpose:** wrap `docs/kashealth-pilot/preflight.sh` as a
first-class skill with auto-fix offers.

**Checks:**

- `gh auth status` scopes (`admin:org`, `project`, `repo`,
  `workflow`)
- Org access (`gh api orgs/<org>` returns 200)
- Org members exist for every `id:` in `.edpa/config/people.yaml`
- Org Issue Types — Initiative, Epic, Story, Defect, Task, Feature
  all present
- `git config user.email` set (auto-commit feature requires it)
- Python version, openpyxl import, ruamel.yaml import

**Auto-fix offers:** for each FAIL, the skill proposes the
exact remediation command and asks the user to approve before
running (per `feedback_skill_first_validation.md` — interactive
flow, not silent execution).

**Where it lives:** `plugin/skills/edpa-preflight/SKILL.md` +
`plugin/edpa/scripts/preflight.py` (port shell logic to Python
for cross-platform; today's `.sh` stays as a no-deps fallback).

#### 2.2.2 Extend `/edpa:setup` — Issue Types check + auto-create

**Today:** `/edpa:setup` runs `project_setup.py` and
`issue_types.py` is a separate manual step (kickoff runbook
§ 1.1).

**v1.10:** `/edpa:setup` first runs `issue_types.py setup --org
<org> --dry-run`, shows the diff, asks confirmation, then runs
without `--dry-run`. Project creation continues only if Issue
Types are present. Eliminates the cryptic GraphQL error when
Issue Types are missing (today's failure mode).

**Implementation:** `plugin/skills/edpa-setup/SKILL.md` gains a
new "Stage 0 — Org-level prerequisites" step before the existing
Stage 1 (people.yaml seed). Underlying Python untouched.

#### 2.2.3 New skill: `/edpa:capacity-override`

**Purpose:** v1.9.0 introduced per-iteration `people:` overrides
in `.edpa/iterations/<id>.yaml`, but the only documented path is
"edit YAML manually + commit + run engine" (runbook § 6.5).
Customer must know schema by heart. Skill wraps that.

**Invocation:** `/edpa:capacity-override <iteration-id> <person-id>
<hours> [--note "<reason>"]`

**Behavior:**

1. Loads `.edpa/iterations/<iteration-id>.yaml`; rejects if
   iteration is already `status: closed` (no retro overrides).
2. Loads `.edpa/config/people.yaml` to validate `<person-id>`
   exists.
3. Computes baseline from `people.yaml`, shows diff
   (`baseline 40h → override 44h, +4h`).
4. Asks for `--note` if not provided (non-trivial overrides
   need audit reason; allow `--note=""` for explicit waiver).
5. Edits the YAML, runs `validate_syntax.py` on the result,
   commits with message `<iteration-id>: capacity override
   <person-id> -> <hours>h`.

**Where it lives:** `plugin/skills/edpa-capacity-override/SKILL.md`
+ `plugin/edpa/scripts/capacity_override.py` (new ~80-line
script).

### 2.3 Runbook trim

Once 2.2.1–2.2.3 ship, `docs/kashealth-pilot/KASHEALTH-PILOT.md`
collapses from 14 sections to ~3 pages. New shape:

1. **Quick orientation** — what the pilot proves
2. **Day-1 setup** — three commands: `/edpa:preflight`,
   `/edpa:setup`, fill `people.yaml`
3. **Weekly cadence** — `/edpa:close-iteration <id>`
4. **PI close** — `/edpa:close-iteration <pi>` + signature
5. **Edge cases** — capacity overrides, A/B simple-vs-gates, rollback
6. **Success criteria + open questions**

The "what skills don't do" annotations vanish — there's no
remaining gap.

## 3. Order of operations

1. **Excel consolidation** (smallest, no skill churn) — ships first as
   1.10.0-alpha. Lets pilot's PI-2026-1 close on v1.9.0 split files,
   PI-2026-2 starts on consolidated.
2. **`/edpa:preflight` skill** — second, because it has no
   dependency on the runbook trim and shortens future debugging.
3. **`/edpa:setup` extension** — third, depends on `issue_types.py`
   already existing (it does).
4. **`/edpa:capacity-override` skill** — fourth, lowest priority
   because the manual YAML path works and the runbook documents
   it. Can slip to v1.10.1 if 1.10.0 needs to ship sooner.
5. **Runbook trim** — last, after all four above land.

## 4. Out of scope

- M365 OOO calendar integration for auto-fill of capacity
  overrides (mentioned in runbook § 14 Q6 as "v2.x roadmap").
- Cost/rate aggregation skill — sazby remain in private registry
  per pilot decision; EDPA doesn't carry rates (memory: `commit
  a4248b2 — drop hourly_rate from public templates`).
- Calibration trigger skill — `/edpa:calibrate` already exists
  via `edpa-autocalib` and runs only when ≥20 ground truth
  records exist; not a gap.

## 5. Acceptance criteria

| # | Criterion | Verification |
|---|-----------|--------------|
| 1 | One `edpa-results.xlsx` per iteration, two tabs, identical content to today's two files | E2E test reads both tabs, asserts row counts match v1.9.0 baseline |
| 2 | `/edpa:preflight` invoked on a clean repo prints actionable FAIL list with auto-fix offers | E2E: install on fresh kas-platform-v1 clone, run, expect Issue Types FAIL with offer to run `issue_types.py setup` |
| 3 | `/edpa:setup` on org missing Initiative/Epic/Story/Defect Issue Types creates them after confirmation, then proceeds without GraphQL error | E2E: revoke 4 Issue Types in test org, run `/edpa:setup`, confirm types created and Project setup succeeds |
| 4 | `/edpa:capacity-override PI-2026-1.3 jurby 28 --note "vacation Jun 9-11"` writes the override, commits, and engine picks it up | E2E: invoke skill, verify YAML diff, run engine, confirm `derived_hours == 28` for jurby |
| 5 | Trimmed runbook is ≤ 3 printed pages | manual page count |
| 6 | E2E suite continues 32+/32+ PASS, all unit tests green | `make e2e && make test` |

## 6. Risks

| Risk | Probability | Mitigation |
|------|-------------|------------|
| Consolidating xlsx breaks downstream Excel macros that target specific filenames | L | none known in pilot or audit-trail; reach-out to PM team if any reference `summary.xlsx` literally |
| `/edpa:preflight` covers too much, becomes its own bug surface | M | port shell verbatim first, no new checks; new checks come in 1.10.x patches with own tests |
| Skill UX collides with non-interactive CI | M | every interactive prompt has `--yes` flag for CI; default is interactive |
| Runbook trim hides useful detail (regret) | L | keep full v1.9.0 runbook as `docs/kashealth-pilot/KASHEALTH-PILOT-detailed.md` archive; new file links it |

## 7. Open questions

1. **Filename for consolidated xlsx:** `edpa-results.xlsx` (this
   RFC) vs `iteration-<id>.xlsx` (matches the `iteration-<id>/`
   directory naming) vs `report.xlsx` (terse). RFC default:
   `edpa-results.xlsx` because it parallels `edpa_results.json`.
2. **Tab order:** Team Summary first or Item Costs first?
   RFC default: Team Summary first (auditor opens to aggregate,
   drills down to per-item).
3. **Should `/edpa:preflight` block on warnings or only on errors?**
   RFC default: WARNING is non-blocking (printed, doesn't exit
   nonzero); ERROR blocks. User confirms.
4. **Should `/edpa:capacity-override` support a `--retro` flag for
   already-closed iterations?** RFC default: no — closed iterations
   are immutable for audit. Retro requires manual `engine
   --re-close` after re-opening, which is a separate skill not in
   v1.10 scope.
