# RFC — v1.10.0 skill-first gaps + Excel workbook consolidation

- **Status:** **proposed** (not implemented)
- **Author:** Jaroslav Urbánek (scope decision 2026-05-07, post-pilot-prep review)
- **Target release:** v1.10.0
- **Required for:** kashealth pilot retro (post-PI-2026-1 close); not blocking pilot kickoff
- **Effort estimate:** ~180 lines code (2 skill extensions + 1 engine refactor) + ~100 lines tests, ~1.5 dev-days

## 1. Why this RFC exists

Pilot kickoff prep (`docs/kashealth-pilot/KASHEALTH-PILOT.md`) revealed
two structural debts that didn't block 1.9.0 release but contradict
the **skill-first validation mandate** (memory:
`feedback_skill_first_validation.md` — *"skill flow je canonical
customer UX, ne script flow"*):

1. The kickoff runbook has 14 sections, of which 9 still call
   `python3 .claude/edpa/scripts/...` directly. That's not a
   documentation problem — it's a **gap inside existing skills**.
   Today's `/edpa:setup` fails with cryptic GraphQL error when
   org Issue Types are missing; there is no skill at all for
   per-iteration capacity overrides (v1.9.0's flagship feature).
2. `summary.xlsx` and `item-costs.xlsx` are split across two
   `Workbook()` instances in `engine.py` for no domain reason —
   it's a code-shape artifact, not a design decision. One file
   with two tabs is the standard audit-deliverable pattern.

Neither item is urgent. Both belong in v1.10.0 as paying down
debt before scaling the pilot to PI-2026-2 / further customers.

## 2. Scope

**Zero new skills.** v1.10 extends two existing skills
(`edpa-setup`, `edpa-close-iteration`) to close the gaps and
refactors `engine.py`'s Excel writer. Adding new top-level
slash commands for "preflight" and "capacity-override" was
considered and rejected — they'd be ceremonies the customer
has to remember instead of natural prep steps inside flows
they already run.

### 2.1 Excel workbook consolidation

**Today:** `engine.py:1126` and `engine.py:1185` build two separate
`Workbook()` objects, written to `summary.xlsx` and
`item-costs.xlsx`.

**v1.10:** one `edpa-results.xlsx` with two tabs:

| Tab | Source | Content |
|-----|--------|---------|
| **Team Summary** | today's `summary.xlsx` | Person · Role · FTE · Capacity · Derived · Items · OK |
| **Item Costs** | today's `item-costs.xlsx` | Item · Level · JS · Person · CW · Score · Ratio · Hours |

**Sketch:**

```python
wb = Workbook()
ws_summary = wb.active
ws_summary.title = "Team Summary"
# … existing summary rows …
ws_items = wb.create_sheet("Item Costs")
# … existing item-cost rows …
wb.save(report_dir / "edpa-results.xlsx")
```

**Migration:**

- Drop both `summary.xlsx` and `item-costs.xlsx` from
  `reports.py` outputs and `.edpa/reports/iteration-*/` layout.
- Replace with single `edpa-results.xlsx`.
- Update fixtures in tests that count xlsx file presence (grep
  `summary.xlsx` and `item-costs.xlsx` in `tests/`).
- Update kickoff runbook § 0 ("Quick orientation") and § A
  ("Pass kritéria po setupu") to reference the new filename.
- Update `plugin/skills/edpa-reports/SKILL.md` output description.

**Backwards compat:** none. v1.10 is a minor bump, customers
re-run engine for new format. Pilot is on v1.9.0 at consolidation
time, so existing kashealth artifacts stay readable as historic
snapshots; only post-v1.10 iterations switch to consolidated file.

### 2.2 Extend `/edpa:setup` — preflight + Issue Types as Stage 0

**Today:** `/edpa:setup` runs `project_setup.py` directly. Org
Issue Types check is a manual step (kickoff runbook § 1.1) and
preflight readiness is an entirely separate shell script
(`docs/kashealth-pilot/preflight.sh`) that today's customer has
to remember to run first.

**v1.10:** `/edpa:setup` gains a **Stage 0 — Org-level prerequisites**
block before today's people.yaml seed step. Stage 0 runs all the
checks that preflight.sh does today plus the Issue Types check,
in one pass, with auto-fix offers for each FAIL.

**Stage 0 checks:**

- `gh auth status` scopes (`admin:org`, `project`, `repo`,
  `workflow`)
- Org access (`gh api orgs/<org>` returns 200)
- Org members exist for every `id:` in `.edpa/config/people.yaml`
  (skipped if people.yaml not yet seeded — Stage 0 can run before
  Stage 1)
- Org Issue Types — Initiative, Epic, Story, Defect, Task, Feature
  all present
- `git config user.email` set (auto-commit feature requires it)
- Python version, openpyxl import, ruamel.yaml import

**Auto-fix flow:** for each FAIL, the skill proposes the exact
remediation command and asks for confirmation before running.
The Issue Types FAIL specifically calls
`issue_types.py setup --org <org> --dry-run`, shows the diff,
asks confirmation, then runs without `--dry-run`. Eliminates
today's "setup fails with cryptic GraphQL error" path.

**Re-runnable:** Stage 0 is idempotent and can be invoked
standalone via `/edpa:setup --check-only` for a customer who
just wants to verify readiness without bootstrapping anything.
That's the customer ask "give me preflight" — without making
preflight its own ceremony.

**Implementation:**

- `plugin/skills/edpa-setup/SKILL.md` gains Stage 0 documentation +
  the `--check-only` flag.
- `plugin/edpa/scripts/preflight.py` (new) ports `preflight.sh`
  logic to Python for cross-platform; existing `.sh` stays as a
  no-deps fallback.
- `project_setup.py` calls preflight.py first, blocks on ERROR,
  prints WARNING.

### 2.3 Extend `/edpa:close-iteration` — capacity overrides as prep step

**Today:** v1.9.0 introduced per-iteration `people:` overrides in
`.edpa/iterations/<id>.yaml`, but the only documented path is
"edit YAML manually + commit + run engine" (runbook § 6.5).
Customer must know schema by heart. There is no skill at all
for this.

**v1.10:** `/edpa:close-iteration` gains a **prep step** before
the engine call. Prep step:

1. Reads `.edpa/iterations/<iteration-id>.yaml` and
   `.edpa/config/people.yaml`.
2. Asks: *"Did anyone have non-baseline capacity this iteration?
   (PTO, sick, overtime, onboarding ramp)"* with `[skip / list
   current overrides / add override]` choices.
3. On `add override`: prompts for `<person-id>` (validated
   against people.yaml), `<hours>` (shows diff
   `baseline 40h → 28h, −12h`), and `<note>` (required if
   non-trivial; allow empty for explicit waiver).
4. Edits the YAML, runs `validate_syntax.py` on the result,
   commits with message `<iteration-id>: capacity override
   <person-id> -> <hours>h`.
5. Loops until user picks `done`, then proceeds to engine call.

**Mid-iteration recording:** for the case "PTO declared Tuesday,
iteration closes Friday — I want to commit the override now",
add a `--prep-only` flag:

```
/edpa:close-iteration PI-2026-1.3 --prep-only
```

Runs only the prep step, skips engine + reports. Same UX as the
full close, just stops after step 4. Closed iterations
(`status: closed`) reject prep-only with "iteration is closed —
overrides are immutable".

**Implementation:**

- `plugin/commands/edpa/close-iteration.md` documents the new
  prep step + `--prep-only` flag.
- `plugin/edpa/scripts/capacity_override.py` (new ~80-line
  helper) handles the YAML edit + validate + commit. Called
  from the close-iteration flow; not exposed as a top-level
  skill.

### 2.4 Runbook trim

Once 2.1–2.3 ship, `docs/kashealth-pilot/KASHEALTH-PILOT.md`
collapses from 14 sections to ~3 pages. New shape:

1. **Quick orientation** — what the pilot proves
2. **Day-1 setup** — `/edpa:setup` (Stage 0 handles all preflight
   + Issue Types + project creation; user fills `people.yaml`
   when prompted)
3. **Weekly cadence** — `/edpa:close-iteration <id>` (prompts
   for capacity overrides as needed)
4. **PI close** — `/edpa:close-iteration <pi>` + signature
5. **Edge cases** — `--prep-only` for mid-iteration recording,
   A/B simple-vs-gates, rollback
6. **Success criteria + open questions**

The "what skills don't do" annotations vanish — there's no
remaining gap.

## 3. Order of operations

1. **Excel consolidation** (smallest, no skill churn) — ships first
   as 1.10.0-alpha. Lets pilot's PI-2026-1 close on v1.9.0 split
   files, PI-2026-2 starts on consolidated.
2. **`/edpa:setup` Stage 0 extension** — second. Highest impact
   on customer UX (cryptic GraphQL error path goes away).
3. **`/edpa:close-iteration` prep step** — third, lowest priority
   because the manual YAML path works and the runbook documents
   it. Can slip to v1.10.1 if 1.10.0 needs to ship sooner.
4. **Runbook trim** — last, after all three above land.

## 4. Out of scope

- M365 OOO calendar integration for auto-fill of capacity
  overrides (mentioned in runbook § 14 Q6 as "v2.x roadmap").
- Cost/rate aggregation skill — sazby remain in private registry
  per pilot decision; EDPA doesn't carry rates (memory: `commit
  a4248b2 — drop hourly_rate from public templates`).
- Calibration trigger skill — `/edpa:calibrate` already exists
  via `edpa-autocalib` and runs only when ≥20 ground truth
  records exist; not a gap.
- Standalone `/edpa:preflight` and `/edpa:capacity-override`
  top-level skills — explicitly rejected, see § 2 framing. The
  use cases are covered by `/edpa:setup --check-only` and
  `/edpa:close-iteration --prep-only` respectively.

## 5. Acceptance criteria

| # | Criterion | Verification |
|---|-----------|--------------|
| 1 | One `edpa-results.xlsx` per iteration, two tabs, identical content to today's two files | E2E test reads both tabs, asserts row counts match v1.9.0 baseline |
| 2 | `/edpa:setup --check-only` on a clean repo prints actionable FAIL list with auto-fix offers | E2E: install on fresh kas-platform-v1 clone, run, expect Issue Types FAIL with offer to run `issue_types.py setup` |
| 3 | `/edpa:setup` on org missing Initiative/Epic/Story/Defect Issue Types creates them after confirmation, then proceeds without GraphQL error | E2E: revoke 4 Issue Types in test org, run `/edpa:setup`, confirm types created and Project setup succeeds |
| 4 | `/edpa:close-iteration PI-2026-1.3 --prep-only` records override interactively, commits, exits without running engine | E2E: invoke skill, answer prompts, verify YAML diff + commit, confirm engine not called |
| 5 | `/edpa:close-iteration PI-2026-1.3` (full close) runs prep step, then engine, picks up override | E2E: continue from #4, run full close, confirm `derived_hours == override` for the person |
| 6 | Trimmed runbook is ≤ 3 printed pages | manual page count |
| 7 | E2E suite continues 32+/32+ PASS, all unit tests green | `make e2e && make test` |

## 6. Risks

| Risk | Probability | Mitigation |
|------|-------------|------------|
| Consolidating xlsx breaks downstream Excel macros that target specific filenames | L | none known in pilot or audit-trail; reach-out to PM team if any reference `summary.xlsx` literally |
| Stage 0 in `/edpa:setup` doubles run time for repeat setups | M | Stage 0 is idempotent and fast (<5 s); cache last-pass timestamp in `.edpa/state/preflight.json`, skip if <5 min old unless `--force` |
| Skill UX collides with non-interactive CI | M | every interactive prompt has `--yes` flag for CI; default is interactive |
| `--prep-only` flag confuses customers ("did it close or not?") | M | output explicitly prints `prep complete — iteration NOT closed; run without --prep-only to close` |
| Runbook trim hides useful detail (regret) | L | keep full v1.9.0 runbook as `docs/kashealth-pilot/KASHEALTH-PILOT-detailed.md` archive; new file links it |

## 7. Open questions

1. **Filename for consolidated xlsx:** `edpa-results.xlsx` (this
   RFC) vs `iteration-<id>.xlsx` (matches the `iteration-<id>/`
   directory naming) vs `report.xlsx` (terse). RFC default:
   `edpa-results.xlsx` because it parallels `edpa_results.json`.
2. **Tab order:** Team Summary first or Item Costs first?
   RFC default: Team Summary first (auditor opens to aggregate,
   drills down to per-item).
3. **Should Stage 0 block on warnings or only on errors?**
   RFC default: WARNING is non-blocking (printed, doesn't exit
   nonzero); ERROR blocks. User confirms.
4. **Should `/edpa:close-iteration` prep step support a `--retro`
   flag for already-closed iterations?** RFC default: no — closed
   iterations are immutable for audit. Retro requires manual
   `engine --re-close` after re-opening, which is a separate
   workflow not in v1.10 scope.
