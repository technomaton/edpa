## Problem

A commit made in iteration N's window that edits a backlog `.md` belonging to a
*different* iteration M (a bulk backlog-authoring commit, or grooming a closed
story) used to get a full-weight `yaml_edit` signal on the foreign item, credited
to the commit author. The engine's credit path is date-blind
(`detect_contributors` sums an item's whole `evidence[]`) and keys off the item's
`iteration:` field, so that weight surfaced in iteration M's report on recompute
and polluted its `contributors[]` snapshot.

## Fix

Write-time guard on both writer paths (`cmd_materialize` + the live post-commit
hook) via the shared `transitions.item_in_iteration` rule: a cross-iteration
`yaml_edit` is recorded but **neutralised to `weight: 0`** (+ `out_of_iteration`
tag) — analytics-only, the same idiom as zero-weight `state_transition`. The
materialize guard targets the iteration being materialized; the hook resolves the
commit's own iteration by author date. Blank-iteration items are left untouched.
Scoped to `yaml_edit`; the analogous full-weight `commit_author` hook leak is
tracked separately (a naive date guard there would drop legitimate
early/spillover delivery work).
