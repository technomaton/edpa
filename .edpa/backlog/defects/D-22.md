pi_planning.py emitted iteration dates only as the pretty `D.M.` string (year dropped by _fmt_date), and the calendar's parseDate fell back to `new Date().getFullYear()` for year-less input — so every PI collapsed onto the current year and PI-2027 iterations painted onto the 2026 month grid (reported with a real board screenshot). Fix: emit authoritative ISO `start_date`/`end_date` fields from load_pis (year-safe; pretty `dates` unchanged for the 4 display consumers), Calendar uses them via iterRange(), and months are grouped into one labeled block per year. Spans Python (pi_planning.py) + TS (types/edpa.ts, server/yaml-store.ts, views/Calendar/Calendar.tsx) + rebuilt vendored pi-bundle.html. 5 regression tests in test_pi_planning_dates.py.
