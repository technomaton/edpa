Windows regression. `_console` (2.1.9) reconfigures stdout to UTF-8 so EDPA's progress glyphs don't crash a cp1250/cp1252 console, but three entry points shipped WITHOUT the guarded `import _console`:

- `pi_planning.py` L331 ‚Äî `print("‚úì ‚Ä¶")`. Reported via `/edpa:pi-planning` on Windows: the run wrote the full ~787 kB report, then exited 1 on `UnicodeEncodeError '‚úì'` in cp1252.py.
- `reconcile.py` L263 ‚Äî `print("‚Ä¶ ‚Äî ‚Ä¶")` (em-dash).
- `transitions.py` L317 ‚Äî `print("‚àÖ")`.

Fix: add the lazy, ImportError-guarded `import _console` at the top of each `main()`. Lazy (inside main, not module-top) because `pi_planning`/`reconcile` are imported by `mcp_server` and must keep JSON-RPC stdout pristine ‚Äî mirrors `create_pi.py`.

Secondary: `pi_planning.py --open` never opened the browser on Windows ‚Äî `subprocess.run(["start", path])` fails because `start` is a cmd builtin, not an executable (FileNotFoundError, swallowed by the bare except). Switched to `os.startfile` on Windows / `open` (mac) / `xdg-open` (linux).

Regression guard: new `tests/test_console.py::test_glyph_printing_entry_points_import_console` AST-walks every entry point and fails if one `print()`s a non-ASCII literal without importing `_console` (mcp_server exempt ‚Äî its glyphs live in JSON-RPC descriptions, framed UTF-8 by the SDK). `test_encoding_hygiene` only guarded file I/O encoding, not stdout ‚Äî that's the gap that let this through.

Verified: reproduced the exact `UnicodeEncodeError '‚úì'`, confirmed fixed; ran all three scripts end-to-end under `PYTHONIOENCODING=cp1252` (exit 0, HTML written); 110 tests pass.
