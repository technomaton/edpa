Windows regression, same family as D-21 (stdout glyph print) ‚Äî the third leg of the locale-encoding trap. Reported from a Czech Windows box running `/edpa:engine`: the run died with

```
Exception in thread Thread-1 (_readerthread):
  File ".../threading.py", line 1082, in _bootstrap_inner
```

The crashing thread is `subprocess`'s stdout/stderr **reader thread**. `subprocess.run(..., text=True)` decodes the child's output with `locale.getpreferredencoding()` ‚Äî **cp1252** on a Czech/German Windows console ‚Äî so the moment `git log` (commit messages, author names) carries a diacritic, the decode raises `UnicodeDecodeError` *inside the reader thread* and the command aborts. Identical to the `text=True` reads in `transitions.py::run_git` and `yaml_edit_signals.py::_run_git` the reporter patched by hand.

`test_encoding_hygiene` already pinned **file I/O** to UTF-8 and `test_console` (D-21) pinned **stdout glyphs**, but neither covered **subprocess text decoding** ‚Äî that's the gap that let this through. Every engine script that shells out to `git`/`gh` and reads the result was affected, not just the two scripts in the report.

## Fix

Add `encoding="utf-8"` to **every** `subprocess.{run,Popen,‚Ä¶}` call that decodes text (`text=True` / `universal_newlines=True`). Found and patched **29 call sites across 20 scripts** via an AST sweep (not grep ‚Äî sees through multi-line calls):

`_auto_commit, _git_timestamps, backlog, capacity_override (√ó3), check_ticket_attached, create_pi, detect_contributors, edpa_commit_info (√ó3), engine, insights, issue_types (√ó2), local_evidence (√ó3), migrate_v1_to_v2 (√ó2), preflight, reconcile, renumber_collisions, sync_pr_contributions (√ó2), transitions, validate_ids, yaml_edit_signals`.

**Deliberately NOT touched:** 6 `capture_output=True` calls that never set `text=` (in `backlog`, `calibrate_signals` √ó2, `reconcile`, `sync_pr_contributions` √ó2). Those capture **bytes** purely to silence git's chatter and never read stdout ‚Äî no decode happens, so they can't hit the trap. Adding `encoding=` there would silently flip the return type `bytes ‚Üí str` and risk breaking byte-consuming callers.

## Regression guard

`tests/test_encoding_hygiene.py::test_all_subprocess_text_pins_utf8` ‚Äî AST-walks every engine script and fails on any `subprocess` call that decodes text (`text=True`/`universal_newlines=True`) without `encoding=`. Bytes-mode `capture_output=True` is correctly NOT flagged (verified: the 6 surviving calls don't trip it). Closes the gap the D-21 closing note already flagged ("test_encoding_hygiene only guarded file I/O encoding, not stdout").

Vendored engine (`.edpa/engine/scripts/`) re-synced so the repo dogfoods the fix.
