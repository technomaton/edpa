`explain.py` (shipped as C1 / D-15) reads `results["derived_reports"]` (match on `r["person"]`) and a flat top-level `results["items"]` (match on `i["contributor"]`, key `job_size`). The engine actually writes `results["people"]` (match on `id`) with **nested** per-person `items` (key `js`, no `contributor`), plus `team_total` / `all_invariants_passed` — there is **no** `derived_reports` and **no** top-level `items`.

Result: `engine.py --explain <person>` / `/edpa:explain` returns `ERROR: person '<x>' not found` against any real `edpa_results.json`. The feature is non-functional in production.

Why 14 unit tests miss it: `test_explain.py` builds a **fictional** `derived_reports`-shaped fixture instead of real engine output — classic "script-pass / skill-fail" gap (docs/E2E-SKILLS-TEST-PLAN.md §0).

Surfaced by the S-238 skill-E2E harness (close-iteration flow) on 2026-06-10.

Fix: explain.py → read `people`/`id`/nested `items`/`js`; use engine `invariant_ok`. Rebuild `test_explain.py` fixture from the real engine schema so the regression is caught.
